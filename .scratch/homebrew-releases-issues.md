## Issue 1: Canonicalize the release identity

### What to build

Make `github.com/flexdinesh/ssh-drop` the single source identity for the module, imports, install documentation, and future release URLs. The project should no longer refer to the pre-release `github.com/dineshpandiyan/ssh-drop` path in code or user-facing install commands.

### Acceptance criteria

- [ ] The Go module path is `github.com/flexdinesh/ssh-drop`.
- [ ] Internal imports use `github.com/flexdinesh/ssh-drop`.
- [ ] README Go install commands use `github.com/flexdinesh/ssh-drop/cmd/ssh-drop`.
- [ ] Existing tests still pass after the module path change.
- [ ] A repository-wide search finds no stale `github.com/dineshpandiyan/ssh-drop` references outside intentionally ignored scratch history.

### Blocked by

None - can start immediately.

## Issue 2: Package `ssh-drop` with GoReleaser snapshots

### What to build

Add GoReleaser configuration and CI validation for `ssh-drop` without publishing stable artifacts yet. GoReleaser should build prebuilt archives for macOS and Linux on `amd64` and `arm64`, stamp the existing CLI version variable, produce checksums, and support snapshot runs on the `dev` branch.

### Acceptance criteria

- [ ] GoReleaser config builds the `ssh-drop` binary from the CLI entrypoint.
- [ ] GoReleaser targets `darwin/amd64`, `darwin/arm64`, `linux/amd64`, and `linux/arm64`.
- [ ] Release archives are named consistently with project, version, OS, and architecture.
- [ ] Release archives include README content.
- [ ] GoReleaser writes `checksums.txt`.
- [ ] Release builds stamp `ssh-drop --version` from the release version.
- [ ] CI runs tests and a normal build on pushes and pull requests.
- [ ] Pushes to `dev` run `goreleaser release --snapshot --clean` without publishing.
- [ ] Local `goreleaser check` succeeds.

### Blocked by

Issue 1.

## Issue 3: Publish stable GitHub releases from manual workflow dispatch

### What to build

Add a manual GitHub Actions release workflow for stable releases from `main`. The workflow should run tests, select the current or next `v0.1.x` tag, create the tag when needed, and run GoReleaser to publish GitHub Release archives and checksums.

### Acceptance criteria

- [ ] The release workflow is manually dispatched.
- [ ] The workflow checks out `main` with full tag history.
- [ ] The workflow sets up Go from the module file.
- [ ] The workflow runs `go test ./...` before tag creation.
- [ ] If `HEAD` already has a matching `v0.1.x` tag, the workflow reuses it.
- [ ] If `HEAD` has no matching tag, the workflow creates the next `v0.1.x` tag.
- [ ] The first release tag is `v0.1.0`.
- [ ] The workflow runs GoReleaser with publishing enabled.
- [ ] The workflow has `contents: write` permission for GitHub Release creation.
- [ ] There is no moving `latest` Git tag.

### Blocked by

Issue 2.

## Issue 4: Open Homebrew tap PRs from GoReleaser

### What to build

Extend the GoReleaser release setup so stable releases generate or update `Formula/ssh-drop.rb` in `flexdinesh/homebrew-tap` by opening a pull request. The formula should install prebuilt release archives, declare `rsync` as a runtime dependency, and verify the installed binary with `ssh-drop --version`.

### Acceptance criteria

- [ ] GoReleaser is configured to publish a Homebrew formula to `flexdinesh/homebrew-tap`.
- [ ] The formula name is `ssh-drop`.
- [ ] The formula homepage points to the canonical `flexdinesh/ssh-drop` repository.
- [ ] The formula installs prebuilt release archives rather than building from source.
- [ ] The formula supports macOS and Linux archives generated by GoReleaser.
- [ ] The formula declares `rsync` as a dependency.
- [ ] The formula test asserts that `ssh-drop --version` includes the formula version.
- [ ] The release workflow uses `HOMEBREW_TAP_TOKEN` for tap access.
- [ ] The tap update opens or updates a pull request instead of pushing directly to tap `main`.
- [ ] The tap branch is deterministic per version, such as `ssh-drop-v0.1.0`.
- [ ] Rerunning a failed release updates the same tap pull request.

### Blocked by

Issue 3.

## Issue 5: Document and verify the release channel

### What to build

Document the stable release process, Homebrew install path, Go install fallback, required GitHub secret, manual release procedure, and local verification commands. Update user-facing install docs so Homebrew is the primary stable path and Go install remains available.

### Acceptance criteria

- [ ] README lists `brew install flexdinesh/tap/ssh-drop` as the primary stable install command.
- [ ] README keeps Go install commands for latest and specific stable releases.
- [ ] Release docs explain the manual workflow and `v0.1.x` tag policy.
- [ ] Release docs explain that there is no moving `latest` Git tag.
- [ ] Release docs describe the required `HOMEBREW_TAP_TOKEN` secret.
- [ ] Release docs describe the generated tap pull request and tap CI handoff.
- [ ] Release docs include local verification commands for tests and GoReleaser validation.
- [ ] Documentation notes that Linux clipboard tooling is optional and separate from the `rsync` dependency.
- [ ] A final local verification run passes tests and GoReleaser validation.

### Blocked by

Issue 4.
